# Decision Log

Concise record of design decisions, kept as written during the build and lightly edited for publication. Status: **RULED** (settled), **PENDING** (awaiting ruling), **TBD** (settled at build time, low stakes).

## Scope and strategy

- **D1. Simple but reliable.** RULED. Small, defensible core; breadth lives in the README security annotations (the project brief invites annotated scope decisions).
- **D2. No token/session issuance.** RULED. This is a credential verification service that owns the auth store; session/JWT strategy belongs to the caller. Documented as a boundary, not an omission.
- **D3. AI use disclosed in README.** RULED. Disclosed by policy; see the README's AI Usage Disclosure section.

## API design

- **D4. Authenticate = `GET /api/v1/users/{username}` + HTTP Basic.** RULED. Authentication demonstrated as authorized access to one's own principal (RFC 7617/7235); the required 200/401 contract is inherited from the RFCs, not invented.
- **D5. Create = `POST /api/v1/users`.** RULED. 201 on success, 409 on duplicate username, 400 on validation failure. JSON in and out.
- **D6. Minimal principal body.** RULED. `{"username": "..."}` only; nothing not required.
- **D7. No RPC-style login alias.** RULED. One authenticate surface; the conventional POST-login alternative is discussed in the README instead of shipped as a second endpoint.
- **D8. Check order on authenticate.** RULED. Missing/unparseable Authorization header: 401 + `WWW-Authenticate: Basic realm`. Header present but username differs from path: 400 before any I/O (no Redis touch, no hashing; pure request-shape rejection, no oracle).
- **D9. Error taxonomy.** RULED. 400 validation/malformed, 401 uniform auth failure, 405 + `Allow`, 409 duplicate, 413 oversize body, 415 non-JSON content type, 429 + `Retry-After` throttle. All errors JSON-bodied.
- **D10. `Cache-Control: no-store` (+ `Pragma`) on auth responses.** RULED. Moving authentication onto GET inherits GET caching semantics; opt-out is explicit.
- **D11. Basic credential parsing splits on the first colon.** RULED. Passwords may contain colons; usernames may not.

## Password policy

- **D12. Standard: NIST SP 800-63B-4 (July 2025), cross-checked against OWASP ASVS 5.0.** RULED. PCI DSS acknowledged and scoped out (no cardholder data in this service's domain). Fact-checked against primary sources.
- **D13. Minimum password length: 15.** RULED. NIST SP 800-63B-4 SHALL for single-factor authentication. Counted in Unicode code points after NFC normalization; generous maximum accepted.
- **D14. Blocklist: in, in the form meaningful at a 15-char floor.** RULED. NIST SHALL satisfied by: structural predicates (all-one-character, repeated block, whole-string sequence/keyboard walk), context containment (username, service name), and a small embedded residual list (canonical corpus head filtered to the length floor, provenance-commented). Pure module, no I/O, no external calls. Each rejection returns a machine-readable code plus a human message naming the rule. HIBP k-anonymity checking (registration-time and login-time recheck, fail-open/fail-closed tradeoff) documented as the production enhancement.
- **D15. No composition rules, no rotation, no truncation.** RULED. NIST SHALL NOTs. Max length generous (>=64 accepted), Unicode counted by code point, NFC normalization.

## Credential storage and verification

- **D16. Hashing: Node core `crypto.argon2` (argon2id).** RULED. OWASP baseline parameters (m=19456 KiB, t=2, p=1), 16-byte random salt, PHC-format stored string. Requires Node >= 24.7.0. No third-party code touches password material.
- **D17. Crypto adapter seam.** RULED. `hash`/`verify` only; the rest of the app never sees algorithm details. Swap path if the core API ever changes.
- **D18. RFC 9106 test vectors in the suite.** RULED. The implementation is verified against the spec at test time.
- **D19. Fail-fast boot capability check.** RULED. Exit with a clear message on Node < 24.7; no silent scrypt fallback (a second hash path is a weaker posture).
- **D20. Uniform 401 + equal-cost work.** RULED. Unknown user and wrong password are indistinguishable in status, body, and timing; the unknown-user path runs one argon2 over the submitted password and discards it, spending the same cost as a real verify. (Originally a verify against a precomputed dummy hash; simplified in layer-3 review to hashing the input directly - same cost, no dummy-hash machinery.)
- **D21. Executable timing invariants.** RULED, **superseded by D50**. Original approach: mock the adapter, assert verify is called exactly once per path. Replaced in layer-3 review — the crypto adapter is no longer mocked/injected (units run real argon2), so the white-box call-count gives way to black-box tests: byte-identical unknown-vs-wrong-password responses plus a relative-timing guard. The 400/missing-header "no store I/O" checks survive via the injected Redis fake.
- **D22. Registration enumeration accepted as inherent.** RULED. Honest 409; the asymmetry (authenticate can be non-enumerating, registration cannot) is documented rather than obfuscated.

## Throttling

- **D23. Per-username failed-attempt throttle in Redis.** RULED, core scope. Checked before any hashing; existence-blind (counters keyed by attempted username regardless of existence); 429 + `Retry-After`. NIST 800-63B-4 throttling SHALL.
- **D24. No in-memory block-cache tier.** RULED. The API tier saturates well before Redis; the economic rejection point for abusive traffic is the edge. Documented as a conditional mitigation if topology ever forces a shared datastore.

## Data layer

- **D25. Redis is the system of record.** RULED. The requirements name "Redis for data storage" verbatim. AOF on, `appendfsync everysec`, hybrid persistence, `maxmemory-policy noeviction` (eviction would be durable deletion, not demotion).
- **D26. Dedicated Redis; the auth store is not shared.** RULED. No other service connects to the credential store. If sharing were ever imposed: ACLs at minimum, and the tradeoff is documented.
- **D27. Username charset constraint.** RULED (pattern set in layer 2). `^[a-z0-9][a-z0-9._-]{2,31}$` — lowercase subset of the POSIX portable-username charset (IEEE 1003.1 "User Name"), first character alphanumeric, 3–32 chars. The charset is the citable part; 32 follows the Linux `useradd` convention, 3 is pure convention (Kirill's call — no standard grounds either bound, and Redis is indifferent: keys cap at 512 MB). Lowercase-only removes case-ambiguity between accounts. Doubles as Redis key-injection defense since the username lands in the key verbatim.

## Runtime, reproducibility, DevOps

- **D28. Demo deliverable is Docker Compose; no cloud deployment.** RULED. App + Redis, pinned exact image versions, healthchecks, `depends_on: service_healthy`, named volume for the AOF, Redis not port-mapped to the host (network isolation over committed-password theater).
- **D29. Reproducible build.** RULED. Exact-pinned dependencies with committed lockfile, `npm ci`, `.nvmrc`, `engines >= 24.7.0`, pinned base image. Unpredictable builds are a security defect.
- **D30. Production shape: this same container on ECS Fargate.** RULED. Dev/prod parity is the image; internal ALB with TLS termination. Compute cost is negligible relative to the datastore.
- **D31. Production datastore: Amazon MemoryDB, not ElastiCache.** RULED. The ruled posture (system of record, noeviction, durability) matches MemoryDB's multi-AZ durable log; ElastiCache is priced and engineered as a cache. Self-managed Redis + AOF noted as the frugal alternative.
- **D32. Lambda considered and rejected.** RULED. Node cold starts are modest (~200-400ms) but land on p99 of an endpoint required to be fast; eliminating them (Provisioned Concurrency) is paying for residency; CPU-proportional pricing forces oversized memory to keep argon2 fast; freeze/thaw suspends connection keepalive machinery, requiring stale-connection recovery a resident container doesn't need. No local Lambda emulation either (parity principle).
- **D33. TLS mandatory in production, documented.** RULED. Demo runs plain HTTP on localhost; the README states the deployment assumption. Internal-network trust carries an explicit zero-trust caveat; service-to-service auth on registration is annotated future work.
- **D37. libuv threadpool identified as the real performance dial.** RULED. Async `crypto` hashing runs on libuv's threadpool (default 4 threads), capping concurrent argon2 hashes per process; JSON handling sits orders of magnitude below hash cost. Demo keeps the default (CPU-bound work gains nothing from threads > cores). `UV_THREADPOOL_SIZE` sizing guidance goes in the README production notes.

## Language and testing

No build step, no `dist/`; what runs is what you read. `tsc --noEmit` runs as a type checker in the test script; `typescript` is a devDependency that never touches the runtime artifact.
- **D39. Test stack: built-in `node:test` + Fastify `inject()` + thin E2E smoke.** RULED. Zero test dependencies. Unit layer: crypto adapter vs RFC 9106 vectors, Basic-auth parser, password policy. Integration layer (the bulk): `inject()` through the real app with adapter and Redis client mocked; carries the D21 invariants. E2E: golden-path smoke over real HTTP against `docker compose up`, gated on an env var so `npm test` needs no Docker.

- **D43. Lint + format: Biome, enforced in CI, no git hooks.** RULED. One devDependency covering both linter and formatter; `npm run lint` gates in CI alongside typecheck and tests. Git hooks rejected as ceremony (and structurally blocked by D40's `ignore-scripts`). Fallback if its floating-promises detection proves immature: typescript-eslint recommended-type-checked set, dev-only, documented.

- **D44. Strict Ajv validation config.** RULED (build finding, layer 1). Fastify's Ajv defaults silently strip unknown body properties (`removeAdditional: true`) and coerce types; both overridden to `false` in the app factory so `additionalProperties: false` actually rejects (400) and type mismatches fail instead of coercing. Also: Fastify ships a built-in `text/plain` parser; removed in the app factory (`removeContentTypeParser`), making the API JSON-only — every non-JSON content type fails negotiation with 415. Remaining D9 taxonomy (error envelope, 404/405+Allow, 413 body-limit tuning, no-store) still lands in its own layer.

- **D45. Error envelope (pulled forward into layer 1).** RULED. Every error response is `{"error": {"code": <machine-readable>, "message": <human-readable>}}` via root-scope `setErrorHandler` + `setNotFoundHandler` (root scope required: Fastify encapsulates error handlers per plugin context). Codes so far: `validation_error`, `unsupported_media_type`, `payload_too_large`, `bad_request`, `not_found`, `internal_error`. 5xx responses are sanitized to a fixed body with the real error logged server-side; a test asserts internals cannot leak. Domain codes (`username_taken`, `invalid_credentials`, `rate_limited`, password policy codes) extend the same mapping in the endpoints layer. Hardened in layer 2: 4xx messages reach callers only from three closed sources — `AppError` (authored), Ajv (derived from the public schema), or a canned table keyed by known framework error codes; unknown 4xx errors keep their status but emit a fixed `bad_request` body (logged as `unmapped 4xx error`), so future dependencies cannot leak messages through the envelope. Test-pinned via an unmapped-418 probe. Completed since: `no-store` headers (D49); 405 + `Allow` and 413 body-limit tuning (D53).

- **D46. E2E harness pulled forward (layer 1).** RULED. `test/e2e.test.ts` probes over real HTTP, env-gated per D39 (`E2E_BASE_URL`; plain `npm test` reports it as skipped). `npm run test:e2e` targets `127.0.0.1:3000`. CI gains the D41 compose-rehearsal job now: checkout → `docker compose up --build --wait` → `test:e2e` (no `npm ci` — the E2E file imports only node built-ins). Currently asserts `/health` only; the golden-path assertions (create user → authenticate) replace it when the real endpoints and Redis land. No Docker-in-Docker needed: GitHub Actions job steps run directly on the runner VM, which ships Docker + compose (the DinD concern was a GitLab-ism).

- **D47. Throttle failure recording: `INCR` + `EXPIRE NX` as one `MULTI`/`EXEC`.** RULED. Recording a failed attempt (D23) is a single atomic batch: `MULTI` → `INCR throttle:<username>` → `EXPIRE throttle:<username> <window> NX` → `EXEC` (one round trip; node-redis `client.multi().incr(k).expire(k, w, "NX").exec()`). The textbook `if count == 1: EXPIRE` client-side branch is rejected: it entrusts the TTL invariant to the one client that won the increment, and a crash in the gap between `INCR` and `EXPIRE` orphans an immortal counter — permanent lockout of that username until manual `DEL`. `EXPIRE ... NX` (Redis >= 7.0: "set the expiry only when the key has no expiry" — condition on the TTL, not on key existence) moves the branch server-side and makes the invariant self-healing: first failure sets the window, later failures no-op, and any orphaned key is repaired by the next failure. Semantics: fixed window from the first attempt in a streak; the alternative (plain unconditional `EXPIRE` = clock restarts per failure, "quiet period required") documented, not chosen. REVISED in layer 4 (D51): the read-only pre-check and the failure record merge into one atomic increment-first gate — `MULTI` `INCR` + `EXPIRE NX` + `TTL` in one `EXEC`, run before hashing, the returned count *is* the gate. This bounds the argon2 a concurrent burst can trigger (a read-only check lets N simultaneous callers pass before any of them records) and counts every attempt with a reset on success (= consecutive failures); `Retry-After` is the returned TTL. Noted: `MULTI`/`EXEC` is atomicity only, no rollback — irrelevant for this batch. Window/threshold values are D36.

- **D48. Layer 2: Redis + registration endpoint.** RULED. `POST /api/v1/users` shipped against Redis (compose service `redis:8.4.4` with the D25 posture flags, named AOF volume, `redis-cli ping` healthcheck, no host port per D28; app gains `REDIS_URL` + `depends_on: service_healthy`). Crypto adapter is hash-only for now — `crypto.argon2` verified present on Node 24.18.0 (RFC 9106 parameter names; returns the raw tag, so the adapter assembles the PHC string); `verify`, the D18 vectors, and the D19 boot check land with authenticate. Full D14 policy shipped: NFC-normalized code-point floor/ceiling (15/512), structural predicates, context containment, residual list (SecLists xato-net top-100k @ `c205c36a`, filtered to >= 15 chars, 70 entries, provenance in-source); the normalized form is what gets hashed. Stored value: JSON document `{"passwordHash": "<phc>"}` at `user:<username>` — Kirill's ruling, extensibility without key migration. 201 carries a `Location` header. Echo endpoint deleted; its negotiation/strict-validation coverage moved onto the users route. `/health` now PINGs the store (sanitized 503 `unhealthy` on failure), so compose `--wait` and the Docker healthcheck transitively prove app↔Redis connectivity. App factory takes injected deps (narrow `RedisLike` seam + hasher; tests fake both). Server connects before listen (fail fast) and closes the client on shutdown. Envelope grows deliberate-error support (`AppError` emitted verbatim; unexpected 5xx still sanitized) and new codes: `username_taken`, `unhealthy`, eight `password_*` policy codes.

- **D49. Layer 3: authenticate endpoint (no throttle yet).** RULED. `GET /api/v1/users/{username}` + HTTP Basic (D4); 200 returns the D6 principal body `{username}`, failures are uniform 401. Two 401 codes (Kirill's ruling): `unauthorized` ("authentication required") when no valid Basic header is presented, `invalid_credentials` ("invalid credentials") when credentials are present but wrong — wrong-user and wrong-password are byte- and timing-identical (D20), verified in-container. Check order (D8): missing/unparseable `Authorization` -> 401 `unauthorized` + `WWW-Authenticate` before any I/O; Basic-username != path username -> 400 `username_mismatch` before any store read or hash (pure request-shape rejection, no oracle). The path `:username` carries no pattern/length schema on purpose — a malformed or absent user falls through to the uniform 401, never a distinct Ajv 400 that would betray structural validity and make authenticate enumerable (D20/D22). Login NFC-normalizes the submitted password through the shared `normalizePassword` (the same form registration hashes) but does not re-run `checkPassword`: the stored password already passed policy, and re-checking would reject grandfathered logins for no security gain. Basic parsing (`src/basic-auth.ts`) is RFC 7617 — case-insensitive scheme, split on the first colon so passwords may contain colons (D11), null -> 401. Crypto adapter gained `verify` (D17): parses the stored PHC and recomputes under the *stored* parameters + salt (so a future cost bump still verifies old hashes), then constant-time `timingSafeEqual`. The unknown-user path spends one verify against a per-process dummy hash (`makeDummyHash`, production cost, computed once at boot, injected as a dep) so presence cannot be timed (D20). D18: the RFC 9106 section 5.3 argon2id known-answer vector is pinned in the suite — Node's `crypto.argon2` accepts `secret` + `associatedData` and reproduces the published tag verbatim, no fallback needed. D19: `assertCryptoCapability(argon2)` at boot refuses to start on a runtime without argon2 (Node < 24.7), no silent fallback. D10: `Cache-Control: no-store` + `Pragma: no-cache` on every authenticate response via a route-level `onSend` (covers a surprise 500 too), plus explicit challenge headers; `WWW-Authenticate: Basic realm="auth", charset="UTF-8"` (neutral realm per D42; charset per RFC 7617). D21 invariants test-pinned: verify runs exactly once on the happy/wrong/unknown paths, and zero verify + zero store touch the 400 mismatch. Envelope vocabulary grows `unauthorized`, `invalid_credentials`, `username_mismatch`. E2E golden path expanded (register -> authenticate 200 + no-store -> wrong password 401 + challenge -> missing header 401), exercising real argon2 verify in the container. No new dependencies. Deferred to the next step: throttle (D23/D36/D47) and 405 + `Allow` (D9) — the handler is ordered so the throttle pre-check (before hashing) and the failed-attempt record slot in without restructuring.

- **D50. Layer-3 review: real-crypto tests, crypto de-injected, black-box timing (amends D21; revises D48/D49 wiring).** RULED. Injecting `hashPassword`/`verifyPassword` purely so tests could observe them was test-induced design pressure. Module mocking was spiked and rejected: it needs `--experimental-test-module-mocks`, fights the ESM module cache (a cached importer keeps its first binding, so per-test mock variation silently fails - proven by spike), and its API is already churning (`namedExports` deprecated to `exports` within Node 24). Resolution: **crypto is no longer injected.** `AppDeps` is `{ redis }` only (Redis stays faked - it is a live resource, and per Kirill's ruling it was left untouched); routes import `crypto.ts` directly; the unknown-user path spends one argon2 over the submitted password and discards it (D20; no dummy hash, no per-process precompute). Unit tests run **real argon2** (~24 ms/op measured on the dev laptop; suite ~0.65s -> ~1.6s), with a shared per-file app fixture (store reset in `beforeEach`; the route-mutating error tests stay per-test) so the app is built once. (Note: removing the dummy hash also removed the per-build argon2 the fixture originally amortized, so the fixture now only saves app reconstruction - keep or drop as taste dictates.) **D21's white-box "verify called exactly once" is replaced by black-box tests**: (1) unknown-user and wrong-password responses are asserted **byte-identical** - the deterministic, always-on guarantee for the content channel; and (2) a **relative-timing guard** for the timing channel - both paths measured on the same machine, interleaved (shared load cancels), reduced with `min` (noise only adds time), asserting the unknown/wrong ratio >= 0.5 (honest ~1.0; a skipped unknown-user argon2 craters to ~0.03), with 3 retries and **no silent skip** (a green means the check ran - a security test that quietly disables itself is false assurance). Calibrated empirically: ratio 0.985-1.032 over 8 runs; a deliberately broken build with the unknown-user argon2 removed was caught (the byte-identical test still passed, the timing test failed), proving the guard has teeth. Absolute times are never asserted (machine-dependent); the ratio is platform-agnostic. HONEST SCOPE, documented for the reader: this crude ratio test is only feasible because the guarded leak is a whole ~24 ms hash op, not a microarchitectural one. **Rigorous timing-side-channel verification uses dedicated tooling** - `dudect` (statistical Welch t-test), `ctgrind`/Valgrind (mark secrets, flag secret-dependent branches), formal ct-verifiers - run as dedicated security testing and out of scope here; the timing guard is a pragmatic regression catch, not a substitute. The "400 mismatch / missing-header touch no store" checks survive via the injected Redis fake (a legitimate dependency boundary, not a crypto internal).

- **D51. Layer 4: per-username login throttle (revises D47's read-only pre-check).** RULED. `GET /api/v1/users/{username}` gained a per-username throttle (D23) between the D8 request-shape checks and any hashing: parse header (401 `unauthorized`, no I/O) -> username mismatch (400, no I/O) -> **`registerAttempt`** -> user lookup + verify. `registerAttempt` is one atomic `MULTI` (`INCR throttle:<u>` + `EXPIRE <u> 900 NX` + `TTL`, one `EXEC`) that counts the attempt and returns the new count; `count > THROTTLE_MAX_FAILURES` (10) -> **429 `rate_limited`** + `Retry-After` (the returned TTL; `Cache-Control: no-store` via the route `onSend`; no `WWW-Authenticate` - a rate limit, not a challenge), skipping hashing. On a rejected verify: just 401 `invalid_credentials` (the attempt already counted itself — no separate record). On success: `resetFailures` (DEL) then 200; counting every attempt with a reset on success is "consecutive failures" (NIST) by another name. **Increment-first, not read-only-pre-check-then-record (the original D47 / first D51 draft):** a read-only check does not bound argon2 under concurrency — N simultaneous requests all read the same sub-limit count and trigger N hashes before any of them records. The atomic increment makes the returned count the exact per-caller gate, so only the first 10 proceed to argon2 and the rest 429, regardless of concurrency — the whole point of gating *before* the hash. (This also retires the "benign race" flagged in the layer-4 self-review; it was not benign.) Threshold `count > 10` lets 10 attempts hash then blocks the 11th; lockout cost is unchanged (still 10 argon2 to lock a name) and key creation stays argon2-bottlenecked (the first attempt on a fresh username is under the limit, so it pays one hash). Existence-blind: keyed by the attempted username before the user lookup, so unknown-user and wrong-password run identical `INCR` + one argon2 + 401 (D20/D50 byte- and timing-identical preserved; the 429 path is identical for known/unknown too). **No new injected dependency (D50):** `src/throttle.ts` (`registerAttempt` / `resetFailures`) operates on the already-faked `redis` seam; `AppDeps` stays `{ redis }`, `app.ts` / `server.ts` unchanged. `RedisLike` grew `incrementCounter` (the atomic `MULTI`, so the gate can never be split from the count) + `clearCounter`. Honest tradeoff vs the read-only design: a locked username now takes a cheap `INCR` write per hammer instead of a read — trivial for Redis (one hot key, O(1)); skipping the increment once over-limit would need a read-first (reviving the race) or Lua, not worth it. Tests are behavioral: N attempts -> 429; a *correct* password is still 429 once tripped (gate precedes verify); an unregistered username throttles identically (no enumeration); a success resets; missing-header and mismatch call `incrementCounter` zero times (ordering). The black-box timing test resets the counter between samples (it hammers one username). E2E trips the throttle against real Redis. Verified live: attempts 1-10 -> 401, 11th -> 429 `retry-after: ~899` + `no-store`, correct-password-while-throttled -> 429. **Documented enumeration edge (reset-on-success):** the 429 is emitted before the user lookup and is byte-identical for existing and non-existent names (test-pinned alongside the 401 byte-identical test), so a *tripped* throttle reveals nothing. Crucially, the reset does NOT let a locked name "resume": once count > 10 the gate precedes verify, so the real user is 429'd too and cannot reset — a locked name stays locked until TTL expiry, and their attempts only push the count higher. The `resetFailures` DEL fires only while count <= 10, where the attacker is seeing 401s regardless, so it is invisible as a state change. The sole residual signal is *deferred lockout*: because the counter is shared, a concurrent successful login can push the attacker's 429 past the 11th attempt, whereas a non-existent or idle name locks deterministically at attempt 11. That leaks "a real login just succeeded on this name" (existence AND active use), not bare existence; it is one-sided (never confirms non-existence), needs a legitimate login to land in the pre-lockout window, and is strictly weaker than the already-accepted registration oracle (D22). Inherent to the D36 reset-on-success ruling; accepted and documented, not mitigated. **Randomizing the threshold or window to blur this was considered and rejected:** window jitter is disclosed by the honest `Retry-After`; threshold jitter (per-username keyed or stochastic) fuzzes the argon2 bound — a reset can defer lockout by ~a full threshold, so masking it needs a jitter range comparable to the threshold, roughly *doubling* the guaranteed hash exposure; and a stochastic gate is either non-sticky (a 429'd attacker just retries until a lucky roll — no real lockout) or needs a stored "locked" flag (added state, and once set the observable is deterministic and still calibratable). Not worth fuzzing the throttle's core bound to blur a signal already weaker than the accepted D22 oracle. Still deferred: 405 + `Allow` (D9); registration / edge volumetric limiting (D24).

- **D52. Production: split the throttle store from the credential store.** RULED (production-only; not in the demo, goes in the README). User hashes and throttle counters want opposite storage policies. The credential store is durable + `noeviction` (D25 — evicting a hash silently deletes an account). Throttle counters are transient (TTL'd), tolerate loss on restart, and want an eviction policy that sheds old keys under pressure (`allkeys-lru` / `volatile-ttl`). Redis sets both persistence *and* `maxmemory-policy` per **instance** (not per key, not per logical DB — `SELECT` shares both), so decoupling the two policies means two instances: the durable credential store (D31 MemoryDB) plus a separate ephemeral cache for the throttle (D31 already frames ElastiCache as "a cache" — exactly the throttle's profile; run it with no AOF/RDB and LRU eviction). The demo keeps a single durable instance per D28: the argon2 cost bottlenecks failed-login volume to ~166/s/instance (D37), so throttle AOF writes and key count stay trivial (~15 MB steady state), and persisting the counter is at worst neutral — a slight plus, since it survives a restart so an attacker cannot clear accumulated failures by forcing a bounce. In-process throttling is not an option (D24: multi-instance needs shared state). Documented in the README security annotations (D1).

- **D53. Layer 5: finish the D9 error taxonomy (405 + `Allow`, 413 body limit).** RULED. **405:** Fastify routes a method-mismatch to `setNotFoundHandler` exactly like an unknown path, with no `Allow` header, and `hasRoute` matches only route *patterns*, not concrete URLs — so `applyErrorEnvelope` builds a `pattern -> methods` table from an `onRoute` hook (which fires for the child-plugin routes because it is registered on the root before the route plugins) and the not-found handler matches the request pathname against it: a hit under other methods -> **405 `method_not_allowed`** + `Allow: <sorted methods>` (includes Fastify's auto-`HEAD` for GET routes); no hit -> unchanged **404 `not_found`**. The pattern->regex covers our static and `:param` segments; a wildcard route (none today) would simply not match and fall through to 404 — safe. **413:** `bodyLimit` set to 16 KiB in the app factory (was Fastify's 1 MiB default); a valid registration body tops out near 6 KiB (32-char ASCII username + 512-code-point password, worst case fully `\u`-escaped astral characters), so 16 KiB leaves margin while cutting the DoS surface 64x; over-limit -> `FST_ERR_CTP_BODY_TOO_LARGE` -> the existing envelope mapping -> **413 `payload_too_large`**. Envelope vocabulary grows `method_not_allowed`. Tests (inject): 405 + exact `Allow` for a wrong method on each route (literal and `:param`), 404 still for unknown / near-miss paths, 413 for an oversize body; verified over real HTTP in compose. This closes the D9/D45 taxonomy (405, 413; `no-store` shipped in D49). Remaining before publication: the README (D1/D3/D42), written last.

- **D54. Redis as an idiomatic decorator plugin; `AppDeps` removed (amends D48/D50; touches D35/D40).** RULED. The injected-options seam (`buildApp(opts, {redis})` threading a one-field `AppDeps` bundle through `register`) is replaced by the Fastify-canonical shape: a `fastify-plugin`-wrapped plugin decorates the instance, and routes read `app.redis`. Rationale: having chosen Fastify we get the most from its conventions — `fastify.redis` is legible to any Fastify reader and keeps us aligned with the ecosystem (decorators + `fp` are how `@fastify/redis` and the Getting-Started DB example share a client), and `AppDeps = {redis}` was a do-nothing wrapper. Structure: new `src/redis/` — `plugin.ts` (owns the narrow `RedisLike` seam + `RedisPluginOptions` + the `fp()` plugin), `fastify.d.ts` (the `declare module "fastify"` augmentation typing `app.redis: RedisLike`; the `import type` above it makes the file a module so the block MERGES rather than replaces). `src/deps.ts` and standalone `src/redis.ts` dissolved into it. **Options mirror `@fastify/redis`:** `{url}` (production — the plugin builds the node-redis client, attaches the `error` listener via `app.log`, decorates the wrapped `RedisLike`, closes it on the `onClose` hook, and `connect()`s during `ready`) or `{client}` (tests — decorate the fake as-is, no I/O). This collapses the connect/onError/close wiring out of `server.ts` (now just: read env, `buildApp({logger:true},{url})`, `shutdown = app.close()`, listen). **Fail-fast preserved:** the plugin connects during `ready`, which `app.listen()` awaits, so an unreachable store still aborts boot. The narrow seam and the sealed atomic `incrementCounter` (D47/D51) are untouched — we decorate the narrow object, never the raw client; `throttle.ts` still takes `redis` as a param (`registerAttempt(app.redis, ...)`). **Config ownership:** `server.ts` (the composition root) keeps reading env (`PORT`, `REDIS_URL`) and passes `{url}` as a plugin option; the plugin never touches `process.env`. Considered `@fastify/env` (Fastify's idiomatic config-as-plugin, schema-validated `fastify.config`) and rejected at our 2-variable scale — an extra dependency that also softens the hermetic-`buildApp` property (tests never read env); revisit if config grows. The `PORT`-NaN validation gap stays open (deferred, cheap to close with explicit lines in `server.ts`, no plugin). **Supply chain (D40):** added `fastify-plugin@6.0.0` (exact) — official Fastify org, authored by a Fastify co-creator, **zero runtime deps**, maintainer set ⊆ `fastify`'s, registry-signed; adds no new trust party (same maintainers whose `fastify` we already run). `@fastify/redis` itself was rejected: it is built on **ioredis**, the client D35 deliberately moved away from (maintenance mode), and it decorates the *full* client (dissolving our seam). **Documented tradeoff:** the augmentation types `redis` as required/ambient on every `FastifyInstance`, so "import the types but forget to register" typechecks and would crash at runtime — the same hole `@fastify/redis` carries. Accepted because we have exactly one composition root that always registers, under test (contained, unlike a published plugin consumed by strangers). Chose required-typing over `redis?` to avoid null-check noise everywhere. This gives up the old compile-time "can't register routes without wiring redis" guarantee, judged cheap in our single-root setup. Verified: typecheck + Biome + 61 tests green (unit suite still only exercises `{client}`); compose both-healthy (real `{url}` connect at boot); E2E 4/4 incl. throttle over real Redis; graceful SIGTERM shutdown proven (`onClose` closes the client, container stops before the SIGKILL timeout); `npm audit signatures` clean incl. `fastify-plugin`. **Renamed after review (same session):** the seam `RedisLike` -> `DataStore`, the decoration `app.redis` -> `app.dataStore`, and `src/redis/` -> `src/data-store/`. Reason: the interface is a *port*, not a subset of the Redis client — `incrementCounter` is a composed `MULTI` transaction, not a command — so it is named for its role, not the backing technology. The injected option branch is now `{store}` (was `{client}`), the export is `dataStorePlugin`, and the throttle/test-helper params follow (`store: DataStore`, `makeFakeDataStore`). `server.ts` is unchanged: it still passes `{url}`, a genuinely Redis connection string for the Redis adapter, and `REDIS_URL` stays the env var name. **Seam return types de-Redis-ised (same review):** `set` no longer takes the always-`{condition:"NX"}` option — it takes `mode: "insert" | "upsert"` and returns `boolean` (whether the write took effect: insert `false` if the key exists, upsert always `true`; the caller uses `false` -> 409). `ping` returns `void` instead of Redis's `"PONG"` (the only caller, `/health`, discards it). `get` keeps `string | null` — that string is the stored payload, not a status word, so it stays. Only `insert` mode has a caller today; `upsert` is implemented (plain `SET`) but unused, the plausible future caller being a password change. **App options are first-class (same review):** `buildApp` takes a single app-owned `AppOptions = { fastifyOptions? } & ({redisUrl} | {store})`, not positional args and not the plugin's `DataStorePluginOptions`. buildApp maps its options to the plugin's at registration (`"store" in options ? {store} : {redisUrl}`), so the plugin's config shape stays out of the app's public signature and the app decides what the plugin gets. `store` is a first-class dependency-injection arm (the analog of `@fastify/redis`'s `{client}`), so the route and pipeline tests reach the real `buildApp` front door with a fake store, with no assembler fork and no code bent around testing. Rejected en route: a `finishApp`/`buildTestApp` split (building production structure around tests), and `mock.module` of the plugin (already rejected in D50: experimental flag, ESM-cache fights, churning API). `server.ts` passes `{fastifyOptions:{logger:true}, redisUrl}`; the test helper passes `{fastifyOptions, store}`.

- **D55. Tree restructure: `src/app/` boundary + features as dependency-declaring plugins.** RULED. Two axes. **(1) Entrypoints vs app:** `src/server.ts` and `src/healthcheck.ts` (the two executables the Dockerfile runs) stay at `src/` root; everything else moved under `src/app/`. `src/app.ts` -> `src/app/index.ts` is the app's public API (exports `buildApp`, `AppOptions`; re-exports `assertCryptoCapability`), so `server.ts` talks only to `./app/index.ts`. The boundary mirrors deployment (two entrypoints, one app), not a features-vs-layers doctrine (there is ~one domain feature and no real layers - routes are thin plugins). Note: `index.ts` does NOT enable a bare `./app` import under our explicit-extension ESM (it is imported as `./app/index.ts`); it is a public-face convention marker only. **(2) Features as plugins:** the auth-domain cluster (`crypto`, `password-policy`, `throttle`, `basic-auth`, the user routes) moved to `src/app/auth/`; health to `src/app/health/`; `data-store/` under `src/app/`; the `routes/` layer bucket is gone. Each feature is a Fastify plugin like `data-store`: `auth/plugin.ts` and `health/plugin.ts` wrap their route plugin in `fp(routes, { name, encapsulate: true, decorators: { fastify: ["dataStore"] } })`. **`encapsulate: true`** (a fastify-plugin option) keeps the feature scoped - unlike `data-store`, which is `fp` WITHOUT encapsulate because it must leak its decorator upward - so `buildApp` composes a uniform list of plugins and is agnostic to whether a plugin is "just routes" or more. **`decorators: { fastify: ["dataStore"] }` encodes the dependency inside the plugin** and Fastify enforces it at boot: proven live - registering `auth`/`health` with no data-store throws `The decorator 'dataStore' required by '<name>' is not present in Fastify`. This closes the D54 hole (the ambient augmentation types `dataStore` as always-present, so TS cannot catch a missing/misordered registration; the decorator check catches it at boot - it validates order, does not reorder, so `buildApp` still registers data-store first). The direct `fp(typedRoutes, ...)` wrap preserves the JsonSchemaToTs type provider (verified: `request.body`/`params` stay typed). The root `onRoute` 405-Allow table still captures routes inside the encapsulated feature plugins (405 tests green). Dockerfile unchanged (`server.ts`/`healthcheck.ts` paths stable; it copies all of `src`). Verified: typecheck + Biome + 61 tests green; boot-enforcement probe; compose both-healthy on the production `redisUrl` path; E2E 4/4; D42 clean. **Amended - `assertCryptoCapability` DROPPED (retires D19's runtime crypto boot-check):** `src/app/boot-check.ts` and its test removed and the check is gone entirely (not relocated). Reason, verified on Node 20.11.1: a named `import { argon2 } from "node:crypto"` **throws at link time** on a runtime without argon2 (`SyntaxError: ... does not provide an export named 'argon2'`, at instantiation, before any module code runs), so `crypto.ts` cannot even load on an incompatible Node - the graceful "requires Node >= 24.7" message could never fire on the exact case it was written for (a namespace import would dodge the link error, but the module-top `promisify(argon2)` would then throw on `undefined` during evaluation, still before any check). The real guards, which cover this without dead code: `engine-strict=true` + `engines.node>=24.7` (npm ci refuses to install on old Node with a clear engines error) and the link error itself (unmistakable at runtime); Docker (`node:24.18.0-slim`) and CI (`.nvmrc`) are always compatible. `server.ts` is now buildApp + listen only; `index.ts` no longer re-exports the check. The one runtime precondition still at the entrypoint is the parked `PORT`-NaN validation.
